Sightline Event Ledger: 4D-FR Architecture

Overview: This schema implements the Four-Dimensional Frame of Reference (4D-FR). It treats the ledger as a Cartesian Database where every transaction is an atomic vector: (Flow, Action, Tail, Head). It includes a "Physics Engine" to prevent fraud by enforcing topological validity (structural coupling).

1. The Lexica (Nodes & Edges)

Defining the valid Nouns (Accounts) and Verbs (Actions) of the system.

-- 1. ACCOUNTS (The Nodes)
CREATE TABLE Accounts (
    account_id INT PRIMARY KEY,
    account_name VARCHAR(100) NOT NULL, -- e.g., "Nvidia H100 Inventory"
    
    -- "Class" defines the network: 'Physical' (Assets) or 'Financial' (Money)
    -- This supports the "Structural Coupling" requirement
    network_type VARCHAR(20) CHECK (network_type IN ('PHYSICAL', 'FINANCIAL', 'BOUNDARY'))
);

-- 2. ACTIONS (The Edges)
-- Captures the "Why". This is the 4th dimension missing in standard GAAP.
CREATE TABLE Actions (
    action_id INT PRIMARY KEY,
    action_name VARCHAR(50) NOT NULL, -- e.g., "Expenditure", "Payment", "Shipping"
    
    -- Defines if this action moves Goods (Irreversible) or Money (Reversible)
    action_type VARCHAR(20) CHECK (action_type IN ('PHYSICAL_FLOW', 'CASH_FLOW', 'COUPLING'))
);

2. The Physics Engine (Topology Rules)

This table acts as the configuration for valid business logic. It dictates which edges can connect which nodes.

-- 3. TOPOLOGY_RULES (The "Physics" Logic)
CREATE TABLE Topology_Rules (
    rule_id INT PRIMARY KEY,
    action_id INT REFERENCES Actions(action_id),
    allowed_tail_type VARCHAR(20), -- Source Network Type
    allowed_head_type VARCHAR(20), -- Destination Network Type
    
    description VARCHAR(255)
);

3. The Event Ledger (4D Transactions)

The core table replacing the General Ledger.

-- 4. EVENT_LEDGER_4D (The Cartesian Database)
CREATE TABLE Event_Ledger (
    event_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    event_date TIMESTAMP DEFAULT NOW(),
    
    -- The 4 Dimensions
    flow_amount DECIMAL(18, 2) NOT NULL,  -- 1: Flow (Quantitative)
    action_id INT REFERENCES Actions(action_id), -- 2: Action (Cause)
    tail_account_id INT REFERENCES Accounts(account_id), -- 3: Tail (Source)
    head_account_id INT REFERENCES Accounts(account_id), -- 4: Head (Destination)

    -- Metadata for Drill-Down (Compositionality)
    unit_type VARCHAR(20), -- e.g., 'USD', 'GPU_UNIT', 'KG'
    sku_reference VARCHAR(50) -- Links to the specific physical object
);

4. The Validator (Fraud Prevention)

A SQL Trigger that rejects any transaction violating the topological rules.

CREATE OR REPLACE FUNCTION validate_topology() 
RETURNS TRIGGER AS $$
DECLARE
    tail_net VARCHAR(20);
    head_net VARCHAR(20);
    is_valid BOOLEAN;
BEGIN
    -- 1. Get the Network Type of the Tail and Head accounts
    SELECT network_type INTO tail_net FROM Accounts WHERE account_id = NEW.tail_account_id;
    SELECT network_type INTO head_net FROM Accounts WHERE account_id = NEW.head_account_id;

    -- 2. Check if this specific Action is allowed to connect these two Network Types
    SELECT EXISTS (
        SELECT 1 FROM Topology_Rules 
        WHERE action_id = NEW.action_id
        AND allowed_tail_type = tail_net
        AND allowed_head_type = head_net
    ) INTO is_valid;

    -- 3. Reject if no rule exists (The "Physics" Violation)
    IF NOT is_valid THEN
        RAISE EXCEPTION 'Topology Violation: Action % cannot connect % to %.', 
            NEW.action_id, tail_net, head_net;
    END IF;

    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER check_topology_before_insert
BEFORE INSERT ON Event_Ledger
FOR EACH ROW EXECUTE FUNCTION validate_topology();
